# YouTube Script Creator (유튜브 대본 제작 스킬)

사용자가 제공한 인기 유튜브 영상의 URL을 분석하여, 그 성공 패턴을 반영한 새로운 대본을 8단계에 걸쳐 체계적으로 작성하는 스킬이다.

## 전체 워크플로우 개요

```
[사용자 입력: 참고 영상 URL + 원하는 주제/방향]
        ↓
   1단계: 성공요인 분석
        ↓
   2단계: 팩트체크
        ↓
   3단계: 추가 정보 검색
        ↓
   4단계: 대본 작성
        ↓
   5단계: 검토 후 수정 (사용자 피드백 반영)
        ↓
   6단계: 스토리보드
        ↓
   7단계: 업로드 정보
        ↓
   8단계: 미디어 생성 스펙 (Flow 이미지 + Grok 영상 YAML)
        ↓
   [최종 산출물: 마크다운 파일로 저장]
```

---

## 시작 전 확인사항

사용자가 이 스킬을 트리거하면, 아래 정보를 확인한다:

1. **참고 영상 URL** — 분석할 유튜브 영상 URL (1개 이상)
2. **새 영상의 주제/방향** — 어떤 주제로 대본을 쓸 것인지
3. **영상 길이** — 쇼츠(~1분) / 일반(5~15분) / 롱폼(15분+) / 미정
4. **채널 성격** — 어떤 채널인지 (정보 전달, 엔터테인먼트, 교육 등)

URL과 주제가 있으면 바로 1단계로 진입한다. 부족한 정보는 진행하면서 자연스럽게 파악한다.

---

## 사전 단계: 유튜브 영상 정보 & 스크립트 수집

유튜브 URL이 주어지면, 아래 순서로 영상 메타데이터와 스크립트를 수집한다.

### 스크립트 수집 — 3단계 우선순위

환경에 따라 가장 효과적인 방법을 자동으로 선택한다:

**🥇 1순위: `youtube-transcript-api` 파이썬 라이브러리**
가장 빠르고 깔끔한 방법. 타임코드 + duration이 숫자로 반환되어 자동화 파이프라인에 직접 투입 가능.

```python
from youtube_transcript_api import YouTubeTranscriptApi

api = YouTubeTranscriptApi()
transcript = api.fetch("VIDEO_ID", languages=["ko"])
for entry in transcript:
    print(f"[{entry.start:.1f}s ~ +{entry.duration:.1f}s] {entry.text}")
```

- `pip install youtube-transcript-api` 필요 (v1.2+)
- 로컬 PC, 자체 서버에서는 정상 동작
- **클라우드 IP(AWS, GCP 등)에서는 YouTube가 차단**하므로 `RequestBlocked` 에러 발생 가능
- 에러 발생 시 자동으로 2순위로 전환

**🥈 2순위: Claude in Chrome 브라우저 도구**
클라우드 환경(claude.ai)에서 1순위가 차단될 때 사용. Chrome 브라우저를 통해 YouTube 페이지에 직접 접근.

1. `tabs_context_mcp`로 탭 준비 → `navigate`로 유튜브 URL 열기
2. `get_page_text`로 JSON-LD schema에서 메타데이터 추출 (제목, 채널, 조회수 등)
3. `javascript_tool`로 "더보기" 클릭 → "스크립트 표시" 버튼 클릭
4. `read_page`로 스크립트 패널의 세그먼트 읽기 (depth=15, max_chars=200000)
   - 각 세그먼트는 `button` 형태로 렌더링됨
   - 내부에 타임코드(`generic "0:00"`)와 텍스트(`text "..."`) 포함
5. 수집된 세그먼트를 `[타임코드] 텍스트` 형식으로 정리

**🥉 3순위: 사용자 직접 붙여넣기**
1, 2순위 모두 실패 시 최후 수단. 사용자에게 아래와 같이 안내:
> "유튜브에서 해당 영상 → 더보기 → 스크립트 표시 → 전체 선택 후 복사해서 여기에 붙여넣어 주세요!"

이와 동시에 `web_search`로 해당 영상의 요약/리뷰/기사를 검색하여 보조 정보를 확보한다.

### 자동생성 자막(ASR) 주의사항

YouTube 자동생성 자막은 음성인식 오류가 많다:
- 고유명사 오류: "이란" → "이단", "밴스" → "빤스" 등
- 전문용어 오류: "핵" → "행", "호르무즈" → "호르몬" 등
- 문맥으로 보정하되, 팩트체크 단계에서 정확한 표기로 최종 수정

### 수집 결과물

수집된 정보를 아래 형태로 정리:

```markdown
## 📋 영상 정보 수집 결과

### 메타데이터
- 제목: [제목]
- 채널: [채널명] (구독자 X명)
- 조회수: [조회수] | 좋아요: [좋아요]
- 길이: [분:초]
- 업로드일: [날짜]
- 태그: [해시태그들]

### 스크립트 (총 XX개 세그먼트)
[0:00] ...
[0:07] ...
(전체 스크립트)
```

---

## 1단계: 성공요인 분석

사전 단계에서 수집한 메타데이터와 스크립트를 기반으로 영상의 성공 요인을 구조적으로 분석한다.

### 분석 항목

#### A. 수치 분석 (웹에서 확인 가능한 범위)
- 조회수, 좋아요 수, 댓글 수
- 업로드 일자 대비 조회수 증가 속도
- 채널 구독자 대비 조회수 비율 (바이럴 지표)

#### B. 콘텐츠 구조 분석
- **후킹 전략**: 첫 3초 / 첫 30초에 어떤 장치를 사용했는가
  - 질문형, 충격적 사실, 반전 예고, 감정 자극, 시각 임팩트 등
- **내러티브 구조**: 어떤 이야기 흐름을 따르는가
  - 문제→해결, 시간순, 비교/대조, 미스터리→해소, 리스트형 등
- **페이싱**: 정보 밀도와 템포 변화
  - 긴장→이완 주기, 장면 전환 빈도, 호흡 조절
- **시청 유지 장치**: 중간 이탈을 막는 전략
  - "잠깐만요", 예고 떡밥, 카운트다운, 퀴즈, 중간 전환점
- **CTA(행동 유도)**: 구독/좋아요/댓글 유도 방식과 타이밍
- **엔딩 전략**: 여운, 다음 영상 예고, 정리 요약

#### C. 톤 & 스타일 분석
- 말투 (존댓말/반말, 격식/비격식)
- 감정 톤 (유머, 진지, 감성, 분석적)
- 타깃 시청자층 추정

### 출력 형식

분석 결과를 아래 형태로 정리하여 사용자에게 보여준다:

```markdown
## 📊 성공요인 분석 결과

### 수치 요약
| 항목 | 수치 |
|------|------|
| 조회수 | ... |
| 좋아요 | ... |
| 업로드일 | ... |

### 핵심 성공요인 TOP 3
1. ...
2. ...
3. ...

### 후킹 전략
- ...

### 내러티브 구조
- ...

### 우리 대본에 반영할 포인트
- ...
```

사용자에게 분석 결과를 보여주고, 대본에 어떤 요소를 반영할지 확인받은 후 2단계로 진행한다.

---

## 2단계: 팩트체크

참고 영상에서 다루는 주요 사실관계와, 새 대본에서 다룰 주제의 핵심 정보를 `web_search`로 검증한다.

### 체크 항목
- 참고 영상에서 언급된 **수치, 통계, 날짜, 인물 정보**의 정확성
- 새 대본 주제와 관련된 **핵심 팩트** 사전 검증
- 출처가 불분명하거나 과장된 주장 식별
- 최신 정보 반영 여부 (오래된 데이터가 아닌지 확인)

### 출력 형식

```markdown
## ✅ 팩트체크 결과

### 참고 영상 팩트체크
| 주장 | 검증 결과 | 출처 |
|------|-----------|------|
| ... | ✅ 정확 / ⚠️ 부분 정확 / ❌ 부정확 | ... |

### 새 대본용 핵심 팩트
| 정보 | 출처 | 신뢰도 |
|------|------|--------|
| ... | ... | 높음/중간/낮음 |

### 주의사항
- 사용하면 안 되는 정보: ...
- 추가 확인 필요: ...
```

---

## 3단계: 추가 정보 검색

대본을 더 풍성하게 만들기 위해, 주제와 관련된 추가 자료를 `web_search`로 탐색한다.

### 검색 방향
- **차별화 포인트**: 참고 영상에서 다루지 않은 각도나 정보
- **최신 동향**: 주제와 관련된 최근 뉴스나 업데이트
- **흥미로운 사례/일화**: 대본에 넣으면 좋을 에피소드, 사례, 비유
- **전문가 의견**: 해당 분야 전문가의 견해나 인용
- **시각 자료 참고**: 영상에서 활용할 수 있는 이미지/그래프 소재 아이디어
- **관련 트렌드**: 유튜브에서 해당 주제가 어떻게 소비되고 있는지

### 출력 형식

```markdown
## 🔍 추가 정보 검색 결과

### 차별화 포인트
- ...

### 최신 동향
- ...

### 활용 가능한 사례/일화
- ...

### 대본 반영 추천 정보
1. ...
2. ...
3. ...
```

사용자에게 어떤 정보를 대본에 반영할지 확인받은 후 4단계로 진행한다.

---

## 4단계: 대본 작성

1~3단계에서 수집한 모든 정보를 종합하여 대본을 작성한다.

### 대본 작성 원칙

1. **후킹은 생명이다**: 첫 문장부터 시청자를 붙잡는다. 1단계에서 분석한 후킹 전략을 참고하되, 그대로 베끼지 않고 변형 적용한다.
2. **구조가 탄탄해야 한다**: 분석된 내러티브 구조를 기반으로 하되, 주제에 맞게 최적의 구조를 선택한다.
3. **팩트가 정확해야 한다**: 2단계에서 검증된 정보만 사용한다.
4. **시청 유지 장치를 배치한다**: 30초~1분 간격으로 궁금증을 유지하는 장치를 넣는다.
5. **톤을 일관되게 유지한다**: 영상 성격에 맞는 말투와 감정 톤을 처음부터 끝까지 유지한다.
6. **자연스러운 말투로 쓴다**: "읽는 글"이 아니라 "말하는 글"로 작성한다. 실제로 소리 내어 읽었을 때 자연스러운지를 기준으로 한다.

### 대본 구조 템플릿

영상 길이에 따라 적절한 구조를 사용한다:

#### 쇼츠 (60초 이내)
```
[후킹 — 1~2문장, 강렬하게]
[본문 — 핵심 정보 3~5개 포인트]
[마무리 — 반전/요약/CTA]
```

#### 일반 영상 (5~15분)
```
[인트로 — 후킹 + 영상 개요 예고] (15~30초)
[섹션 1 — 도입/배경 설명] (1~3분)
[전환 장치]
[섹션 2 — 핵심 내용] (2~5분)
[전환 장치]
[섹션 3 — 심화/사례/반전] (2~4분)
[아웃트로 — 정리 + CTA + 다음 영상 예고] (30초~1분)
```

#### 롱폼 영상 (15분+)
```
[프리 후킹 — 가장 자극적인 장면/정보 미리보기] (10~20초)
[인트로 — 주제 소개 + 영상 로드맵] (30초~1분)
[챕터 1~N — 각 챕터별 소주제] (각 3~5분)
  ├ 챕터 내 미니 후킹
  ├ 본문
  └ 다음 챕터 예고
[클라이맥스 — 핵심 메시지/결론] (2~3분)
[아웃트로 — 요약 + CTA] (30초~1분)
```

### 대본 작성 형식

```markdown
## 🎬 대본

**영상 제목(안):** ...
**예상 길이:** ...분
**톤:** ...

---

### [인트로] ⏱️ 0:00~0:30

**화면 설명:** ...

**내레이션:**
> "여기에 실제 대본 내용을 작성합니다..."

---

### [섹션 1: 섹션 제목] ⏱️ 0:30~3:00

**화면 설명:** ...

**내레이션:**
> "..."

*💡 시청 유지 장치: ...*

---
(이하 반복)
```

각 섹션에는 반드시 **화면 설명**과 **내레이션**을 함께 쓴다. 시청 유지 장치가 들어가는 곳에는 표시를 해둔다.

---

## 5단계: 검토 후 수정

작성된 대본을 사용자에게 보여주고 피드백을 받는다.

### 자체 검토 체크리스트 (사용자에게 보여주기 전에 확인)

- [ ] 후킹이 충분히 강력한가?
- [ ] 팩트가 모두 정확한가? (2단계 결과와 대조)
- [ ] 시청 유지 장치가 적절한 간격으로 배치되어 있는가?
- [ ] 톤이 일관되는가?
- [ ] 자연스럽게 읽히는가? (문어체가 아닌 구어체인가)
- [ ] 영상 길이에 맞는 분량인가?
- [ ] CTA가 자연스럽게 들어가 있는가?
- [ ] 참고 영상의 성공요인이 잘 반영되었는가?

### 사용자 피드백 수집

대본과 함께 체크리스트 결과를 보여주고, 아래 사항을 확인받는다:
- 전체적인 방향과 톤이 맞는지
- 수정하고 싶은 부분이 있는지
- 추가하거나 빼고 싶은 내용이 있는지
- 특정 표현이나 문장에 대한 의견

피드백을 받으면 수정본을 작성하고, 사용자가 만족할 때까지 반복한다.

## 6단계: 스토리보드

확정된 대본을 기반으로 **제작 파이프라인(TTS → 자막 → 이미지 → 영상 → 편집)**에 직접 투입할 수 있는 수준의 스토리보드를 제작한다.

### 제작 파이프라인 이해

스토리보드의 최종 소비자는 "사람의 눈"이 아니라 **자동화 파이프라인**이다:
```
대본 → TTS 음성 생성 → 자막 타이밍 추출
              ↓
스토리보드 컷 단위 → AI 이미지 생성 → AI 영상 생성(I2V)
              ↓
자막 + 영상 클립 + BGM → 편집/이어붙이기 → 롱폼 완성
```

따라서 스토리보드는 **컷 단위까지 촘촘하게** 설계해야 한다. "대략 이런 느낌"이 아니라, 각 컷의 이미지 프롬프트와 영상 프롬프트를 바로 복사해서 쓸 수 있어야 한다.

### 컷 밀도 기준

시청자의 시각적 집중력과 제작 효율을 고려한 경험적 공식:

| 영상 유형 | 컷 체인지 간격 | 1분당 컷 수 |
|-----------|--------------|-----------|
| 쇼츠 (1분) | 2~4초 | 15~30컷 |
| 일반 (5~15분) | 5~8초 | 7~12컷 |
| 롱폼 (15분+) | 5~10초 | 6~10컷 |

> **기본 공식: 1분당 약 7컷** (뉴스/분석형 기준)
> 스토리텔링형은 1분당 10컷, 쇼츠는 1분당 20컷까지 올릴 수 있다.

### 화면 유형 4분류

모든 컷은 아래 4가지 유형 중 하나로 분류한다:

| 유형 | 설명 | 비율 (권장) | 제작 방식 |
|------|------|-----------|----------|
| **🎬 AI영상** | 핵심 장면, 동적 요소가 필요한 컷 | 20~30% | Image-to-Video 생성 |
| **🖼️ 정적이미지+모션** | AI 이미지에 켄번즈(줌/팬) 적용 | 35~45% | 이미지 생성 후 모션 편집 |
| **📝 텍스트오버레이** | 핵심 키워드, 자막 강조, 인용문 | 15~25% | 편집 소프트웨어에서 제작 |
| **📊 인포그래픽** | 지도, 차트, 비교표, 다이어그램 | 10~20% | 편집 소프트웨어에서 제작 |

> AI 이미지가 필요한 컷 = AI영상 + 정적이미지+모션 = 전체의 약 55~75%
> AI 영상화 컷 = 전체의 약 20~30% (제작 시간/비용 절감)

### 컷 수 예측표 (스토리보드 작성 전에 반드시 먼저 제시)

대본이 확정되면, 스토리보드 상세 작성 전에 아래 예측표를 먼저 사용자에게 보여주고 확인받는다:

```markdown
## 📐 컷 수 예측

| 항목 | 수치 |
|------|------|
| 영상 총 길이 | X분 (X초) |
| 평균 컷 간격 | X초 |
| **총 컷 수** | **약 XX컷** |
| AI 이미지 필요 | 약 XX장 (전체의 XX%) |
| AI 영상 필요 | 약 XX개 (전체의 XX%) |
| 텍스트/인포그래픽 | 약 XX개 (전체의 XX%) |
| 예상 이미지 생성 시간 | 약 XX분 |
| 예상 영상 생성 시간 | 약 XX분 |

### 섹션별 컷 분배

| 섹션 | 길이 | 컷 수 | 이미지 | 영상 | 그래픽 |
|------|------|------|--------|------|--------|
| 인트로 | Xs | X | X | X | X |
| 섹션1 | Xs | X | X | X | X |
| ... | ... | ... | ... | ... | ... |
| **합계** | | **XX** | **XX** | **XX** | **XX** |
```

### 3레벨 스토리보드 구조

스토리보드는 3단계 계층으로 구성한다:

```
Level 1: 섹션 (Section) — 2~5분 단위 — 대본의 큰 주제 구분
  └ Level 2: 시퀀스 (Sequence) — 20~60초 단위 — 하나의 소주제/논점
      └ Level 3: 컷 (Cut) — 3~10초 단위 — 하나의 시각적 장면 (제작의 최소 단위)
```

### Level 3 컷(Cut) 상세 형식

각 컷에는 아래 정보를 모두 포함한다. 이 형식은 자동화 파이프라인에서 직접 사용할 수 있도록 설계되었다:

```markdown
#### Cut 001 | ⏱️ 0:00~0:05 | 🎬 AI영상

**내레이션:**
"호르무즈 해협을 지나는 유조선 한 척, 한 척마다"

**화면 유형:** 🎬 AI영상
**카메라 모션:** 서서히 줌인 (위성 시점 → 유조선 클로즈업)

**이미지 프롬프트 (영문):**
> "Aerial satellite view of Strait of Hormuz, single oil tanker passing through narrow waterway between Iran and Oman, deep blue ocean, photorealistic, cinematic composition, golden hour lighting, 16:9"

**영상 프롬프트 (I2V용, 영문):**
> "Camera slowly zooms in from satellite altitude toward the oil tanker. The tanker moves steadily from left to right through the strait. Gentle ocean waves. Duration: 5 seconds."

**자막 오버레이:** "호르무즈 해협" (하단 중앙, 흰색, 반투명 배경)
**BGM/효과음:** 긴장감 있는 저음 드럼 시작
**전환:** 없음 (첫 컷)
```

### 카메라 모션 가이드

컷의 화면 유형에 따라 적절한 카메라 모션을 지정한다:

| 화면 유형 | 권장 모션 | 설명 |
|----------|----------|------|
| 🎬 AI영상 | I2V 프롬프트에 명시 | 실제 카메라 움직임 + 피사체 동작 |
| 🖼️ 정적이미지+모션 | 켄번즈 효과 지정 | 줌인/줌아웃/팬(좌→우, 우→좌)/틸트(상→하, 하→상) |
| 📝 텍스트오버레이 | 텍스트 애니메이션 | 페이드인/타이핑/스케일업 |
| 📊 인포그래픽 | 순차 등장 | 항목이 하나씩 나타나는 방식 |

### 캐릭터/스타일 일관성

채널에 고유 캐릭터나 시각 스타일이 있다면, 모든 이미지 프롬프트에 공통 스타일 태그를 붙인다:

```
[공통 스타일 태그 예시]
"Pixar-inspired 3D style, soft lighting, warm color palette, consistent character design: [캐릭터 설명]"
```

이 태그는 스토리보드 상단에 한 번 정의하고, 각 컷의 프롬프트에서 참조하도록 한다.

### 최종 출력물

스토리보드 완성 후 아래 순서로 출력한다:

1. **컷 수 예측표** (전체 요약)
2. **섹션별 시퀀스/컷 목록** (전체 구조 한눈에)
3. **컷별 상세 스토리보드** (제작에 바로 투입 가능)
4. **이미지 프롬프트 일괄 목록** (이미지 생성 파이프라인용)
5. **영상 프롬프트 일괄 목록** (I2V 파이프라인용)

## 8단계: 미디어 생성 스펙

6단계 스토리보드의 컷 정보를 기반으로, **Flow 이미지 + Grok 영상** 자동 생성 파이프라인에 직접 투입할 수 있는 YAML 스펙을 생성한다.

### 파이프라인 연동 구조

```
Stage 6 스토리보드 (컷별 이미지/영상 프롬프트)
        ↓ 변환
Stage 8 미디어 생성 스펙 (통합 YAML)
        ↓
  ┌─────┴─────┐
  ▼           ▼
[Flow]      [Grok]
이미지 생성   영상 생성 (reference_image: auto로 Flow 이미지 참조)
```

### YAML 포맷

스토리보드 이후 아래와 같은 YAML 블록을 `## Stage 8: 미디어 생성 스펙` 헤딩 아래에 생성한다:

````markdown
## Stage 8: 미디어 생성 스펙

```yaml
defaults:
  flow:
    model: nano_banana_pro         # imagen_4 | nano_banana_pro | nano_banana_2
    aspect: "16:9"                 # 16:9 | 4:3 | 1:1 | 3:4 | 9:16
    count: 1                       # 1~4 (생성 후 택 1)
    resolution: "2K"               # 1K | 2K
  grok:
    resolution: "720p"             # 480p | 720p
    duration: "6s"                 # 6s | 10s
    aspect: "16:9"                 # 2:3 | 3:2 | 1:1 | 9:16 | 16:9
    cooldown_min: 30               # 생성 간 최소 대기 (초)
    cooldown_max: 40               # 생성 간 최대 대기 (초)
  output_dir: "./output/{project_name}/"

media:

  - cut: 1
    time: "0:00-0:05"
    type: infographic
    flow:
      prompt: "Split screen infographic, left side showing '6 items' in green, right side '24 items' in red, bold typography, clean white background, graphic design style"
      model: imagen_4               # defaults와 다를 때만 명시
      filename: "주제_cut01_infographic"
    grok:
      prompt: "Numbers animate onto split screen with impact, left side items appear one by one, right side items flood in chaotically, camera holds steady, dramatic reveal"
      reference_image: auto         # 같은 cut의 Flow 이미지 자동 매칭
      filename: "주제_video_cut01"

  - cut: 2
    time: "0:05-0:15"
    type: person
    flow:
      prompt: "Person sitting on couch scrolling smartphone, warm living room lighting, casual setting"
      filename: "주제_cut02_person"
    grok:
      prompt: "Person on couch scrolling phone with frustrated expression, camera slowly zooms in on phone screen, then person sighs"
      duration: "10s"               # defaults와 다를 때만 명시
      reference_image: auto
      filename: "주제_video_cut02"

  - cut: 3
    time: "0:15-0:20"
    type: abstract
    flow:
      prompt: "Abstract illustration of multiple forking paths, muted colors with one highlighted path, minimalist style"
      filename: "주제_cut03_abstract"
    grok: null                      # 이미지 + 편집 줌으로 충분

  - cut: thumbnail
    type: thumbnail
    flow:
      prompt: "YouTube thumbnail, dramatic split comparison, bold text space in center, high contrast, 1280x720"
      model: imagen_4
      aspect: "16:9"
      filename: "주제_thumbnail"
    grok: null
```
````

### 컷 유형(type) 분류

각 컷에 `type` 필드를 부여하여 프롬프트 스타일을 결정한다:

| type | 설명 | 프롬프트 스타일 |
|------|------|-------------|
| `infographic` | 차트, 수치, 타이포그래피 | "bold typography", "clean design", "data visualization" |
| `person` | 인물, 행동, 표정 | 자연스러운 장면 묘사, 조명/표정 포함 |
| `abstract` | 추상 일러스트, 과학적 시각화 | "illustration", "artistic style", 개념적 표현 |
| `location` | 장소, 배경, 환경 | 구도+분위기, "wide angle", "cinematic" |
| `thumbnail` | 썸네일 전용 | "YouTube thumbnail", 고대비, 1280x720 |

### 이미지(Flow) vs 영상(Grok) 프롬프트 작성 규칙

| 항목 | 이미지 (Flow) | 영상 (Grok) |
|------|-------------|------------|
| 핵심 | 구도, 색감, 스타일 | 동작, 변화, 카메라워크 |
| 추천 키워드 | "clean design", "bold typography", "cinematic composition" | "camera zooms in", "slowly appears", "animates", "pans across" |
| 금지 키워드 | "animated", "motion", "moving" (정적 이미지이므로) | 제한 없음 |
| 길이 | 1~2문장 | 2~3문장 (동작 시퀀스 묘사) |
| 언어 | 영문 | 영문 |
| 참조 | 독립적 (자체 완결) | `reference_image: auto`로 Flow 이미지 참조 |

### `reference_image` 동작 방식

| 값 | 동작 |
|----|------|
| `auto` | 같은 cut의 Flow 이미지 자동 매칭 (`output_dir/{flow.filename}.png`) |
| `"path/to/file.png"` | 특정 파일 경로 지정 |
| 키 없음 또는 `null` | 프롬프트만으로 생성 (참조 이미지 없이) |

### `grok: null` 판단 기준

모든 컷에 영상이 필요한 것은 아니다. 아래 기준으로 `grok: null` 여부를 결정한다:

| 장면 유형 | 영상 필요? | 판단 근거 |
|----------|----------|----------|
| 인포그래픽 (숫자 애니메이션, 차트 변화) | O | 데이터가 시각적으로 변하면 영상 |
| 인포그래픽 (정적 차트/표) | X → `grok: null` | 이미지 + 편집 줌/팬으로 충분 |
| 인물 (동작, 표정 변화) | O | 스크롤, 걷기, 표정 등 동적 요소 |
| 인물 (호스트 정적 샷) | X → `grok: null` | 이미지 + 미세 줌으로 충분 |
| 추상 (변화, 흐름) | O | 시냅스 활동, 데이터 흐름 등 |
| 추상 (정적 일러스트) | X → `grok: null` | 이미지 + 편집 효과로 충분 |
| 장소 (패닝/달리) | O | 카메라 움직임이 필요한 장면 |

> 일반적으로 전체 컷의 약 60%가 영상, 40%가 이미지만 사용.

### 모델 선택 가이드 (Flow)

| 장면 유형 | 권장 모델 | 이유 |
|----------|----------|------|
| 인포그래픽/타이포그래피 | `imagen_4` | 텍스트 렌더링 정확도 우수 |
| 인물/크리에이티브 | `nano_banana_pro` | 사실적 인물, 자연스러운 라이팅 |
| 추상/아트 | `nano_banana_pro` | 예술적 표현력 우수 |
| 장소/배경 | `nano_banana_pro` | 풍경/실내 사실감 |
| 썸네일 | `imagen_4` | 텍스트+고대비 필요 |

### 파일명 규칙

```
{프로젝트명}_cut{번호:2자리}_{type}.png     # Flow 이미지
{프로젝트명}_video_cut{번호:2자리}.mp4       # Grok 영상
{프로젝트명}_thumbnail.png                  # 썸네일
```

- 프로젝트명은 한글 가능 (예: `선택역설`)
- cut 번호는 01부터 시작, 2자리 zero-padding
- 썸네일은 별도 `cut: thumbnail`로 관리

### 생성 순서

1. **Phase 1: Flow 이미지 생성** — `media[].flow` 파싱하여 전체 이미지 배치 생성
2. **Phase 2: Grok 영상 생성** — `media[].grok` 파싱 (`grok: null` 제외), Phase 1 이미지를 `reference_image: auto`로 참조

Phase 2는 반드시 Phase 1 완료 후 실행한다 (참조 이미지 의존).

### Stage 8 생성 시 주의사항

- Stage 6 스토리보드의 컷 번호, 타임코드와 **정확히 1:1 대응**시킨다
- 이미지 프롬프트에 "animated", "motion" 등 동적 키워드를 절대 사용하지 않는다
- `defaults`와 동일한 값은 각 cut에서 생략하여 YAML을 간결하게 유지한다
- `defaults`와 다른 값만 개별 cut에 명시한다 (예: `duration: "10s"`, `model: imagen_4`)
- 각 프롬프트는 **영문**으로 작성한다 (AI 이미지/영상 생성 모델의 영문 최적화)
- 파일명은 **한글+영문 혼용** 가능

---

## 최종 산출물

모든 단계가 완료되면, 전체 내용을 하나의 마크다운 파일로 정리하여 `/mnt/user-data/outputs/`에 저장한다.

파일명 규칙: `[주제]_유튜브대본_YYYYMMDD.md`

파일 구조:
```markdown
# 🎬 유튜브 대본: [주제]
> 생성일: YYYY-MM-DD
> 참고 영상: [URL]

## 1. 성공요인 분석
(1단계 내용)

## 2. 팩트체크
(2단계 내용)

## 3. 추가 정보
(3단계 내용)

## 4. 대본
(4단계 내용)

## 5. 스토리보드
(6단계 내용)

## 6. 업로드 정보
(7단계 내용)

## 7. 미디어 생성 스펙
(8단계 YAML 블록)
```

`present_files` 도구를 사용하여 사용자에게 파일을 전달한다.

---

## 주의사항

- 각 단계가 끝날 때마다 사용자에게 결과를 보여주고, 다음 단계로 넘어갈지 확인받는다. 단, 사용자가 "한번에 쭉 해줘" 등 일괄 처리를 원하면 중간 확인 없이 진행한다.
- 참고 영상의 대본을 그대로 베끼지 않는다. 구조와 전략을 참고하되 내용은 독창적으로 작성한다.
- 팩트체크에서 불확실한 정보는 반드시 표시하고, 사용자에게 직접 확인을 권한다.
- 저작권에 주의한다. 참고 영상의 문구를 그대로 가져오지 않는다.
- 영상 길이를 모르는 경우, 주제와 정보량을 고려하여 적절한 길이를 제안한다.
